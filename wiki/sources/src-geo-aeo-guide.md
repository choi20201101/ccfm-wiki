# src-geo-aeo-guide

> **GEO·AEO 실행 가이드** (현장 미팅 녹취 기반, 외부 검색 미포함).
> GEO = Generative Engine Optimization (생성형 엔진 최적화), AEO = Answer Engine Optimization (답변 엔진 최적화).
> AI(LLM)가 내 콘텐츠를 **인용·노출**하게 만드는 실전 방법. 일부는 발화자 경험 기반 관찰이라 검증 여지 있음.

## 핵심 원리 — 역순 설계
LLM은 **학습 데이터(RAG)에서 1차 답변 → 웹서치로 보완하며 출처를 붙인다.** 봇이 사이트를 긁는 행위 자체가 인용·학습.
한 번 학습·인용되면 **그다음엔 다시 안 가져갈 수도 있다.** → 모든 실행의 출발점: "AI가 굳이 내 콘텐츠를 긁어 노출시켜줄 이유"를 만드는 것 = **명확하게 좋은 정보(권위 + 정확도)**.

## 1. 사이트 세팅 (기술 기반)
- **스키마·데이터 구조를 '오늘의집'처럼** (커머스 1순위). 오늘의집은 쿠팡급 인용량 — 데이터 구조·스키마가 잘 짜여서. SEO 전문가에게 "오늘의집 구조 적용" 요청하거나 구조를 긁어 같은 형태로 교체. **단, 검증 필수** (AI가 100% 완성 안 해줌).
- **도메인 지수 누적** — AI는 도메인 지수를 실제로 많이 평가. 도메인 나이 + 외부 링크 연결 + 지속 운영. 단기 최적화 불가, **시작이 빠를수록 유리**.
- **SEO 노출 버블 열기** — 크롤링 막히지 않게 열어두면 봇 유입. 초반에 많이 가져감.
- **이미지 신중 선정** — "봇은 이미지 못 읽으니 alt만"은 틀림. 비전 모델이 이미지 내용까지 인식. alt에만 의존 금지.

## 2. 콘텐츠 작성
- **역순 설계** — "AI가 왜 내 걸 노출시켜줄까" → '명확하게 좋은 정보'. 어설픈 콘텐츠는 안 가져감.
- **실제 사람이 쓴 글 + 실제 사진** — 자동화로 긁어가게 해도 AI 글로 인식돼 걸러짐 = 인용 아님. 프리랜서 작가 등 실제 작가 + 직접 찍은 사진 필요. **함정**: 맡긴 작가가 다시 LLM으로 쓸 수 있음 → 신뢰 가능한 사람 / AI 글 필터 장치 필요.
- **LLM별 글 유형 패턴화** — Claude·Gemini·GPT가 긁어가는 글 유형이 다름. 타깃 모델에 맞춰 작성 패턴 준비.

## 3. 키워드 전략
- **대화형 문장 키워드(롱테일)를 대량으로.** 이제 단어가 아닌 문장형 질문(예: "두피 문신 하고 싶은데 M자 탈모가 있어서…"). 세부까지 다 잡으면 **100만 개 규모**, 각각 월 1회만 노출돼도 누적 유입이 큼. 메인 키워드는 경쟁 세고 전환 약함 → **세부·롱테일을 폭으로**. 키워드별 콘텐츠를 **자동화로 월 단위 지속 발행**.

## 4. 플랫폼별 접근
- **Gemini** — 구글 모델이라 **구글 SEO 영향 최대**. 구글 SEO·스키마 충실 = 곧 GEO.
- **Perplexity** — 웹서치 의존 높음 → 새 콘텐츠 비교적 빨리 반영.
- **Gemini** — 웹서치 의존 낮고 **캐시·반영 기간 가장 김**(업데이트 느림). 한 번 자리잡으면 오래 감. ⚠️ *발화자 체감 관찰, 모델 업데이트로 변동 가능.*
- **네이버에 갇힌 경우(스마트스토어)** — 네이버가 막아둬 GEO 잘 안 됨. → **외부에 스키마 완비된 클론/허브 랜딩** 만들어 우회. 클론에서 서브·메인 키워드 모두 노출 → 트래픽 모이면 스마트스토어로 넘김 → **UTM·커스텀 추적**으로 전환 확인.

## 5. 측정 & 반복
- ⚠️ **측정은 API·스크린 모드로** — 결과값은 로그인·개인화에 따라 사람마다 다름. 광고주가 본인 PC로 검색하면 본인 사이트 안 뜸. API·스크린 모드로 지문 보내 나오는 값 기준. "안 보인다"가 곧 "실패"는 아님(압도적이지 않으면 노출 자체가 안 됨).
- **GA로 LLM 유입·키워드 추적** — 어떤 LLM이 유입됐는지 + 키워드 리퍼러로 잡힘.
- ⚠️ **봇 트래픽 ≠ 좋은 노출** — 한 사이트 누적 544만 중 절반이 AI 봇. 봇 방문은 학습·인용이고 한 번 학습되면 다음엔 안 가져갈 수도. 봇 유입량만으로 "노출 잘 된다" 판단 금지.
- **노출 키워드 리포트 → 다음 주제 결정 → 반복** — 실제 노출 키워드를 기간별 리포트로 정리하고 "다음 콘텐츠 주제"까지 도출해 재발행. 이 루프를 계속.

## 6. 주의사항 (도메인 사망 리스크)
- ⚠️ **구글은 어뷰징 판단 시 도메인 영구 사망**(복구 안 됨). 무리한 물량·어뷰징 = 도메인 통째 상실 위험. → **정확하게 쓰는 편이 오히려 빠르고 안전.**
- ⚠️ **어뷰징 한계는 '내구 테스트'로만** — 버려도 되는 테스트 사이트에서만 한계 확인. 메인 자산으로 실험 금지.
- ⚠️ **자동화 콘텐츠만으론 한계** — 자동화는 키워드 커버리지·발행 효율까지만. 인용 노리는 핵심 콘텐츠는 사람이 직접 = **이원화 필수**.

## 향후 적용 (블로그/워드프레스/구글 글 발행 자동화)
이 가이드는 **발행 자동화 설계의 기준 프레임**이다. 자동 발행 파이프라인 구축 시:
- 자동화 = 롱테일 키워드 커버리지·발행 효율 담당 (대량 문장형 키워드 × 월간 발행)
- 인용 노림 핵심 콘텐츠 = 사람 글 + 실제 사진으로 이원화
- 스키마(오늘의집형) + 도메인 지수 누적 + 크롤 가능 상태를 전제로
- 발행 후 GA·API/스크린 모드로 LLM 유입·노출 키워드 측정 → 다음 주제 도출 루프
- 어뷰징 회피(정확성 우선)로 도메인 사망 방지
